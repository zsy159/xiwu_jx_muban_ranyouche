# 薪酬流水线观察台 — 界面设计文档

> 版本：v0.1  
> 日期：2026-06-23  
> 状态：草案（待评审）  
> 关联：`main.py` CLI、`output/<月>/reports/`、`salary_pipeline/validation/parity.py`

---

## 1. 背景与痛点

### 1.1 当前观察方式

| 动作 | 现状 | 不便之处 |
|------|------|----------|
| 跑批 | `python main.py compute / compute-all --reconcile` | 终端输出碎片化，需记住子命令 |
| 对账结论 | `output/<月>/reports/差异报告_*.md` | 需手动找最新文件；整体通过/未通过不直观 |
| 差异定位 | Markdown 表格 + JSON | 无法按店别/职务/列快速筛选；难以对比 golden vs computed |
| 公式告警 | `formula_warnings*.txt` | 与对账报告分离，不知道影响哪些行 |
| 金标准引导 | 散落在代码与 WORKLOG | 无法一眼看出「哪些列仍借 Excel」 |
| 多账套 | 提成汇总 / 售后 / XW发薪 | 三套报告格式相近但需分别打开 |

### 1.2 核心用户

| 角色 | 目标 | 频率 | 优先级 |
|------|------|------|--------|
| **财务 / 业务验收**（主展示对象） | 看当月各表是否对齐、哪岗有差异、能否放心用系统数 | 每月 1–2 次 | **P0** |
| **迭代开发者** | 改引擎 → 跑批 → 下钻到公式/列 | 每天多次 | P0（功能完整） |
| **运维 / 实施** | 接入新月份 raw 数据、切换账期 | 每月 1 次 | P1 |

**财务向设计原则**：默认隐藏拓扑/公式告警等技术细节；用语「金标准 / 系统计算 / 差异」；支持导出简短验收摘要（PDF/打印友好 MD）。

### 1.3 设计目标

1. **一眼看健康度**：当月各锚点表对账通过与否、未通过岗位数、告警数。  
2. **三步下钻**：表 → 岗位 → 人/单元格（golden / computed / Δ）。  
3. **零复制路径**：从差异格一键跳到拓扑公式、源 sheet（后期）。  
4. **贴合现有产物**：优先读 `差异报告_*.json`、`formula_warnings*.txt`，不重复造对账逻辑。  
5. **本地优先**：数据敏感，默认单机运行，不上云。

## 2. 头脑风暴：产品形态

### 2.1 关键问题拆解

观察不便的本质不是「少一个网页」，而是缺少 **运行态 + 对账态 + 追溯态** 的统一视图：

```text
运行态   哪条 pipeline 跑了？何时？产出路径？
对账态   哪张表、哪个岗位、哪一列、差多少？
追溯态   这格公式是什么？是否金标准引导？源数据在哪？
```

### 2.2 方案对比

| 维度 | A. Streamlit 观察台 | B. FastAPI + 轻量前端 | C. 增强 CLI（Rich/TUI） |
|------|---------------------|------------------------|-------------------------|
| 开发成本 | 低（1–2 天 MVP） | 中（1–2 周） | 极低（半天） |
| 表格/筛选 | 中等，可嵌 AgGrid | 强，完全自定义 | 弱 |
| 触发跑批 | 可 subprocess 调 main.py | 同左 + WebSocket 日志 | 天然支持 |
| 部署 | `streamlit run` 本地 | 需前后端构建 | 无 |
| 扩展性 | 中期够用 | 长期最好 | 不适合复杂下钻 |
| 与项目栈 | Python 单体，契合 | 需引入 TS 工具链 | 纯 Python |

**推荐：A → B 演进**

- **MVP（迭代 5）**：Streamlit「观察台」，只读 `output/` + 可选触发 CLI。  
- **Phase 2**：若需多人协作或复杂 diff UI，再拆 FastAPI + React，JSON 契约不变。

### 2.3 命名与定位

**产品名（工作名）**：**薪酬流水线观察台**（Payroll Pipeline Console）

定位：**开发期的对账驾驶舱**，不是替代 Excel 的业务录入界面。

| UI 标签 | CLI | JSON 报告 | 告警文件 | 产出 xlsx |

### 4.3 差异探索 Diff Explorer

**目的**：类似 Excel 并排对比，支持筛选与导出；**财务默认看「验收列」**，开发可展开更多列。

#### 列显示策略（问题 3 的结论）

| 模式 | 列范围 | 谁用 | 说明 |
|------|--------|------|------|
| **验收列**（默认） | `month.yaml` → `parity.columns` / `payout_parity.columns` 等 | **财务** | 与 CLI `reconcile` 判定一致：Hub 为 F–P（11 列），XW 为 9 列核心发薪指标。界面「通过/未通过」与终端一致，不产生歧义。 |
| **绩效块** | Hub 的 W–AI（整车绩效、加装绩效…） | 开发 / 迭代 | 迭代 4+ 正在补齐；部分仍金标准引导。财务模式下默认**折叠**，避免看到大量 NaN 误以为系统坏了。 |
| **全部业务列** | 导出 xlsx 中的全部指标列 | 开发 | 手动开启；加载较慢，用于排查导出列缺失。 |

**为何财务默认不用 W–AI？**

- `reconcile` 目前只验收 F–P（见 `parity.columns`），财务关心的「这月账能不能关」以这套列为准。  
- W–AI 很多格仍依赖金标准 Excel，并排对比会出现「系统列为空、金标准有数」——对财务像 bug，对开发是预期的过渡期状态。  
- 双模式避免财务被未验收列干扰；开发用侧边栏切换到「绩效块 / 全部列」即可。

```text
列显示  ○ 验收列（推荐，财务）  ○ 含绩效 W–AI  ○ 全部列
```

```text
┌─────────────────────────────────────────────────────────────────┐
│ 差异探索   表: 提成汇总   模式: [仅差异列] [仅未通过岗位]          │
│ Join: 店别 + 职务 + 姓名                                         │
├──────────────────────────┬──────────────────────────────────────┤
│ 左侧：行列表              │ 右侧：列对比（选中行）                  │
│ □ 彭州 出纳+内勤 牟春柳   │  列名      golden   computed   Δ      │
│ □ …                      │  考核量     100      100       0      │
│                          │  保险毛利    0.0      NaN       —  ⚠   │
│                          │  整车绩效   1200     1180      -20      │
└──────────────────────────┴──────────────────────────────────────┘
```

**实现要点**

- 后端按需加载：`read_golden_summary_sheet` + `read_computed_summary_excel`（已有）。  
- 大表分页：默认只加载 `parity.columns` 配置列 + 用户追加列。  
- 单元格样式：`|Δ| > tolerance` 高亮；`NaN` 单独样式（金标准引导缺失 vs 公式失败）。  
- 导出：当前筛选结果 → CSV。

## 5. 交互流（用户故事）

### US-1：迭代开发者查失败原因

1. 打开观察台 → Dashboard 见「提成汇总 ⚠」。  
2. 进入对账中心 → 筛选「仅未通过」→ 点击「出纳+内勤」。  
3. 看列级差异 → 点「保险毛利」样例 → 跳转差异探索。  
4. 并排查看 golden/computed → 复制 join key 到 IDE 调试。  
5. 切告警中心看同行是否有 `#REF!` 或 `eval fail`。

### US-2：验收迭代关闭

1. Dashboard 四张锚点卡片全 ✅。  
2. 打开 Golden Map 确认计划内引导项已标注。  
3. 导出对账摘要 PDF/MD（一键生成，内容同现有报告但更短）。

### US-3：端到端试跑

1. Run 面板执行 `compute-all --reconcile`。  
2. 日志完成后 Dashboard 自动更新。  
3. 对比「使用金标准 hub」与「合并计算 hub」的 XW 发薪差异（并排两次 run manifest）。


**UI：「导入其他月份」**（MVP 可先做注册 + 目录检测，上传放 Phase 2）

| 步骤 | 行为 |
|------|------|
| 检测 | 扫描 `data/raw/*`、`output/*` 子目录，与 registry 合并 |
| 注册 | 用户填写月份 ID、选择 raw 目录路径 → 写入 registry |
| 校验 | 检查三套 xlsx 是否存在；无 `output` 时 Dashboard 显示「仅有金标准，未跑批」 |
| 切换 | 全局状态 `current_month` 切换，重载对应该月的 config 与 reports |

**Phase 2 可选**：上传 zip（raw + 可选 output）到 `data/import/<月>/`，后台解压并注册——接口在 `observability/loaders.register_month()` 预留，MVP 不实现上传 UI。

**跨月对比**（非 MVP）：两月 parity 摘要并排（通过率、未通过岗位数趋势），依赖 registry 有多个月且均有 reports。

### 7.1 栈

| 层 | 选型 | 理由 |
|----|------|------|
| UI | **Streamlit 1.x** | 纯 Python、表格/图表够用、本地启动快 |
| 表格 | `st.dataframe` / **streamlit-aggrid** | 排序、筛选、高亮 |
| 数据 | 现有 `parity.py` 加载函数 + JSON | 不重复对账 |
| 跑批 | `subprocess` + `main.py` | 复用 CLI |
| 配置 | `month.yaml` + `observability.yaml` | 与流水线一致 |

### 7.2 目录建议

```text
salary_pipeline/
  observability/
    __init__.py
    loaders.py          # 发现最新报告、读 manifest
    models.py           # UI 用 view model（薄封装 ParityReport）
    golden_map.py       # 读 observability.yaml
  app/
    streamlit_app.py    # 入口
    pages/
      1_总览.py
      2_对账中心.py
      3_差异探索.py
      4_告警.py
      5_金标准引导.py
```

启动：

```bash
streamlit run salary_pipeline/app/streamlit_app.py
```

| 项 | 要求 |
|----|------|
