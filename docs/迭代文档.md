## 1. 项目目标（共识）

**不是**一次性把 5 月 Excel 算一遍，**而是**：

1. 以 **2026 年 05 月** 三套 Excel 为**样板月（金标准）**；
2. **反推**其中的计算公式与表间依赖；
3. 将公式沉淀为 **可复用的 Python 计算流水线**；
4. 未来每月将 Excel 放入 `data/raw/YYYY-MM/`，更新 `config/month.yaml` 后执行 `python main.py compute` 产出当月结果。

核心原则：

| 原则 | 说明 |
|------|------|
| 公式驱动 | 以 Excel 公式拓扑为依据写代码，而非仅凭业务口述 |
| 样板月对账 | 每个计算阶段必须与 5 月 Excel 列级/单元格级比对，通过后才进入下一迭代 |
| 数据 / 规则 / 逻辑分离 | 当月明细是输入；`提成依据` 是规则；`ops/` + `pipelines/` 是逻辑 |
| 向量化优先 | `SUMIF` → `groupby`；`IF` 阶梯 → `np.select` / `pd.cut`；避免无意义逐行循环 |

### 2.3 公式函数分布（指导算子优先级）

**销售账套（西物）Top 函数：** `IF`(34k)、`SUMIF`(25k)、`INDEX`/`MATCH`(各 20k)、`COUNTIF`(14k)、`SUMIFS`(6k)

**售后账套（吉利）Top 函数：** `SUMIF`(5.5k)、`INDEX`/`MATCH`(各 3.4k)、`SUM`(3.3k)

**规则库 Top 函数：** `IF`、`SUM`、`SUMIFS`

→ 第一版算子库应优先覆盖：**SUMIF、INDEX+MATCH、IF/IFS、SUM、SUMIFS、COUNTIF**。

## 5. 计算链路（业务视角）

### 5.1 销售端（西物）

**设计原则（2026-06-23 修订）：表分两层——`提成汇总` 是汇总枢纽（hub），各「发」表才是最终发薪表（output）。**

```text
明细层（input — 当月原始数据，可导入）
  终端明细表、整车成本、按揭明细、保险明细、装饰台账 …
        ↓
中间公式层（formula — 由模块重算）
  销售任务及完成率、绩效整理表、渗透率 …
        ↓
汇总枢纽层（hub — CommissionSummaryBuilder）
  提成汇总   ← 跨岗位/跨店聚合；禁止从 Excel 硬性导入
        ↓  主方向：约 55%「发」表公式格 SUMIF 从此表取数
        ↓  例：XW!F3 = SUMIF(提成汇总!D:D, D3, 提成汇总!F:F)
最终发薪层（output — 各店/渠道发薪表）
  XW提成-发、CS提成-发、直营店提成-发、XW提成-翼真发 …
        ↓
对账比对（reconcile — 开发期）
  ① 计算版 提成汇总  vs  金标准（枢纽层对账）
  ② 计算版 XW提成-发 等  vs  金标准（发薪层对账 — 业务终验收）
```

**Excel 中的双向引用（需代码侧处理）：**

| 方向 | 量级（XW 示例） | 含义 |
|------|-----------------|------|
| `提成汇总` → `XW提成-发` | XW 约 1565 个公式格引用汇总 | **主数据流**：发薪表从汇总表取考核量、提成项 |
| `XW提成-发` → `提成汇总` | 汇总约 281 个公式格回引 XW | 汇总表 BB/BC 等列再收各店发薪合计；与上形成**环** |

```bash
# 生产模式：模块计算 → 汇总枢纽 → 发薪表
python main.py compute

# 对账模式（当前骨架先对 hub；发薪表对账迭代 2+ 扩展）
python main.py reconcile

# 依赖闭包（两个锚点含义不同）
python scripts/closure_report.py --anchor 提成汇总    # 往上追：算汇总需要哪些表
python scripts/closure_report.py --anchor XW提成-发   # 往上追：算发薪表需要哪些表（含汇总枢纽）
```

| 表角色 | `sheet_registry.yaml` | 说明 |
|--------|----------------------|------|
| `input` | 终端明细表、整车成本… | 每月替换的原始数据 |
| `formula` | 销售任务及完成率、绩效整理表… | 中间表，由模块重算 |
| `hub` | **提成汇总** | 汇总枢纽；供各「发」表取数；**不是**最终发薪表 |
| `output` | **XW提成-发**、CS提成-发… | 各店/渠道**最终发薪表** |

**对账验收口径：**

- 关联键：`店别` + `职务` + `姓名`
- 按**职务（岗位）**分组比对所有共有列
- 某岗位 `mismatch_cells == 0` → 该岗位计算逻辑通过
- **枢纽层**（提成汇总）与**发薪层**（各「发」表）分别对账；发薪层为业务终验收

**闭包锚点（销售）：**

| 锚点 | 闭包方向 | 用途 |
|------|----------|------|
| `提成汇总` | 往上追依赖 | 规划中间模块、输入表接入顺序 |
| `XW提成-发` 等 | 往上追依赖 | 确认发薪表上游是否含汇总枢纽及明细链 |

**典型公式：**

- 汇总枢纽：`提成汇总!F3 = SUMIF(销售任务及完成率!C:C, D3, 销售任务及完成率!Y:Y)`
- 发薪表取数：`XW提成-发!F3 = SUMIF(提成汇总!D:D, D3, 提成汇总!F:F)`

### 5.2 售后端（吉利）

```text
产值/明细层
  5月车间产值、05保明细、保险专员5月外拓 …
        ↓
业务/前台层
  05月业务提成（SA：产值 × 完成率阶梯系数）
        ↓
门店汇总层
  05月提成-武侯售后、05月提成-机场售后 …
        ↓ INDEX+MATCH → 05基本（补贴 lookup）
```

**输出锚点（售后）：`05月提成-武侯售后`**（可先做一个门店，再复制到机场）

- 典型公式：`=INDEX('05基本'!I:I, MATCH(C5, '05基本'!C:C, 0)) + …`

### 5.3 规则库（提成依据）

- 6 张 sheet：销售提成标准、售后工资提成标准、邀约专员提成等
- 多为**岗位说明 + 人员/阶梯**，不宜整表硬编码；迭代 3 抽取为 `config/commission_rules/*.json`

### 5.4 月度生产时什么变、什么不变

| 每月变化 | 相对稳定 |
|----------|----------|
| 各明细表数据行 | 公式结构（算子） |
| 人数、台次、金额 | 表间依赖拓扑 |
| 工作簿文件名 / sheet 名中的月份 | 输出表列语义 |
| 偶发规则调整（`提成依据`） | `sheet_registry` 可配置 |

## 8. 公式模式 → 算子映射表（实施参考）

| Excel 模式 | Python 实现 | 优先级 |
|------------|-------------|--------|
| `SUMIF(表!K:K, 键, 表!V:V)` | `df.groupby(键)[V].sum()` + `map`/`merge` | P0 |
| `SUMIFS(...)` | `groupby` 多条件或 `pd.merge` 后 `sum` | P0 |
| `INDEX(列, MATCH(键, 键列, 0))` | `merge(on=键)` 或 `set_index().reindex` | P0 |
| `IF(a<>0, IF(b/a>cap, cap, b/a), 0)` | `np.where` 嵌套 | P0 |
| `IF(α<0.85, 0, IF(α<1, α*0.015, 0.018))` | `np.select` / `pd.cut` | P1 |
| `A1+A2+…+An`（同行多格求和） | `df[cols].sum(axis=1)` | P1 |
| `COUNTIF` / `COUNTIFS` | `groupby().size()` 或布尔聚合 | P1 |
| 跨表整列引用 `表!K:K` | 读整列 DataFrame，注意性能 | P1 |

**不逐格翻译 9 万个公式**；按列识别模式，一列一个算子调用。

## 9. 对账（parity）规范

### 9.1 开发期（样板月 2026-05）

- **列级：** 输出表每一列与 Excel `read_data` 结果 `assert_series_equal`（或误差阈值）
- **单元格级：** 拓扑中标记为「高价值」的格（汇总行、合计、实发）
- **日志：** 每次 `merge` / `groupby` 后打印 `shape`、主键唯一性计数

### 9.2 生产期（其他月份）

- 无金标准时：总额校验 + 抽样人工复核 + 人数/台次环比异常检测
- 新月份首次上线：建议保留一个月与旧 Excel 并行双跑
