## 1. 读本文档前先知道的三件事

### 1.1 系统输出是什么

| 项 | 内容 |
|----|------|
| 行数 | 473（2026-05 样板月） |
| 列数 | **27 列**（4 键列 + 23 值列，含 AT/BC） |
| 粒度 | 订单级（一行 ≈ 一笔 VIN / 服务补录） |
| 下游 | `HubFormulaEngine` 按 `P(销售顾问)` 做 `SUMIF` 汇总进提成汇总 W–AI |

### 1.2 不读什么

- **不**从金标准 **绩效整理表** sheet 拷贝任何数值列（`GOLDEN_OVERLAY_COLUMNS = ()`）。
- 金标准整理表仅用于 `parity_report` 列级对账测试。
- **终端明细表** 已登记 ingestion，但 **BC（终端返利）** 尚未实现，当前 25 列不依赖该表。

### 1.3 数据从哪来

所有明细仍来自 **同一份** 月度销售工作簿（`month.yaml` → `workbooks.sales`），读的是 **input 表**，不是财务已算好的整理表结果。

另有一份 YAML 配置参与骨架构建：

- `salary_pipeline/config/performance_sheet_columns.yaml`（账期、顾问别名、补录行）

## 2. 构建流水线

```text
系统销售毛利（筛选 + 去重）
        ↓
订单骨架 O / G / P / K
        ↓
┌───────────────────────────────────────────────────────┐
│ Slice 1–3：明细直算                                      │
│   保险明细 → AB/AJ/AO/AP                                 │
│   按揭明细 + 按揭原表 → AK/AL                            │
│   装饰台账 → BH                                          │
│   整车成本 → AW–BB                                       │
└───────────────────────────────────────────────────────┘
        ↓
┌───────────────────────────────────────────────────────┐
│ Slice 5：闭包列                                          │
│   enrich_order_context（中间字段 H/I/R/L/S/A/D/V…AA）   │
│   compute_closure_columns → AG/AH/AI/AM/AN/AS/AQ/AR   │
└───────────────────────────────────────────────────────┘
        ↓
computed_perf_frame（25 列）→ 导出 Excel / 注入 Hub 公式引擎
```

## 4. 列级溯源总表（来源精准到列）

**格式说明：** `源表!列(表头)`；关联键写在「关联」列。

| 列 | 中文表头 | 关联键 | 来源（精准到列） |
|----|---------|--------|-----------------|
| **G** | 订单号 | — | `系统销售毛利!B(订单号)`；筛选 `!BA(结算日期)=账期` 且 `!AZ(订单类型)=含整车订单` |
| **O** | VIN码 | — | `系统销售毛利!BD(VIN码)`；同上筛选；VIN 去重 keep first |
| **P** | 销售顾问 | — | `系统销售毛利!BJ(销售顾问)`；11 个 VIN → YAML `p_overrides_by_vin` |
| **K** | 台数 | — | 默认 `1`；`supplement_rows` 补录行 K 可为空 |
| **AB** | 保险返利收入 | O | `SUMIF(保险明细!D(车架号), O, 保险明细!BP(返利金额))` |
| **AJ** | 保险提成 | O | `SUMIF(保险明细!D(车架号), O, 保险明细!BS(提成金额))` |
| **AK** | 按揭提成 | O | `SUMIF(按揭明细!G(VIN码), O, 按揭明细!BO(基础绩效))` |
| **AO** | 座位险绩效 | O | `SUMIF(保险明细!D(车架号), O, 保险明细!BU(座位险提成))` |
| **AP** | 玻璃险绩效 | O | `SUMIF(保险明细!D(车架号), O, 保险明细!BV(玻璃险提成))` |
| **AL** | 盈利产品 | O | `SUMIF(按揭原表!AC(车架号), O, 按揭原表!AF(盈利性奖励))` + `SUMIF(按揭明细!G(VIN码), O, 按揭明细!BR(吉致额外追加奖励))` |
| **BH** | 装饰成本 | G | `SUMIF(装饰台账!N(订单号), G, 装饰台账!AK(含税成本价))` |
| **AW** | 出厂指导价/扣除降价补差 | O | `INDEX(整车成本!R(出厂指导价), MATCH(O, 整车成本!K(全车架号), 0))` |
| **AX** | 提车现返 | O | `INDEX(整车成本!S(提车现返), MATCH(O, 整车成本!K, 0))` |

| **AZ** | 项目金额附加费 | O | `INDEX(整车成本!U(项目金额附加费), MATCH(O, 整车成本!K, 0))` |
| **BA** | 广告返利 | O | `INDEX(整车成本!V(广告返利), MATCH(O, 整车成本!K, 0))` |
| **BB** | 提车奖励当月返 | O | `INDEX(整车成本!W(提车奖励当月返), MATCH(O, 整车成本!K, 0))` |
| **AG** | 单台绩效 | O,K | 见 [§5.1 AG](#51-ag-单台绩效) |
| **AI** | 加装绩效 | O,G,K | 见 [§5.2 AI](#52-ai-加装绩效) |
| **AH** | 整车超额 | O | 见 [§5.3 AH](#53-ah-整车超额) |
| **AM** | 爱车宝 | O | `SUMIF(爱车保!F(VIN), O, 爱车保!BA(爱车宝提成金额))` |
| **AN** | 上户绩效 | O | `SUMIF(上户提成!B(VIN码), O, 上户提成!H(提成金额))` |
| **AS** | 置换服务 | O | `SUMIF(置换服务!G(VIN), O, 置换服务!BB(提成金额))` |

## 5. 闭包列 — 中间字段与公式链

闭包列在内存中先构造中间字段（**不写入导出 Excel**），再算出 AG/AH/AI 等。

### 5.0 闭包共用中间字段

| 中间字段 | 来源（精准到列） | 用途 |
|---------|-----------------|------|
| **H** 车种 | `系统销售毛利!D(车种)` by VIN | AG、AQ 查提成标准 |
| **I** 销售渠道 | `系统销售毛利!E(销售渠道)` by VIN | AG、AQ；派生 D |
| **R** 部门 | `系统销售毛利!BL(部门)` by VIN | → 比对表 → A |
| **A** 指标汇总部门 | `比对表!A(系统部门)` → `比对表!B(指标汇总部门)` | AG、AQ；派生 D |
| **D** 渠道 | 代码派生：`A` + `I`（直营店/二网规则） | AH 分支判断 |
| **L** 订单合计 | `系统销售毛利!AO(订单合计(含税))` by VIN | AI |
| **S** 精品最低价 | `系统销售毛利!BQ(精品最低价金额)` by VIN | AI |
| **U** 不提成精品 | `装饰台账!AR(最低价含税金额)` 按 `!H(精品名称)` + `!N(订单号)`/`!M` 规则汇总到 G | AI |
| **V** 整车节约 | `INDEX(系统超额!P(合计超卖金额), MATCH(O, 系统超额!W(VIN码), 0))` | 权限链 |
| **Z** | `SUMIF(按揭原表!AC(车架号), O, 按揭原表!AN(整车超权限))` | 权限链 |
| **X** 经理权限 | 硬编码 `0`（样本月 58 行常数 1000 未建模） | 权限链 |
| **Y** | V + X | 权限链 |
| **AA** 实际整车节约权限 | 若 Z≠0 则 0，否则 Y−Z | AH |

### 5.1 AG 单台绩效

1. `系统销售毛利!BL` → `比对表!A→B` 得 **A**
2. 取 **H**=`!D`、**I**=`!E`、**K**
3. `提成标准!C(部门)+!D(销售渠道)+!E(车型)` 匹配 → `提成标准!F(2026年标准) × K`
4. 特例：A=武侯自有店 且 H=星越L → `200 × K`

### 5.2 AI 加装绩效

1. **L**=`系统销售毛利!AO`，**S**=`!BQ`，**U**=装饰台账规则汇总（见上表）
2. K=0 → `L × 12%`
3. K≠0 且 (S−U)>0 → `(S−U) × 12%`；否则 0

**装饰台账 U 规则（按订单号 G 汇总 AR）：**

| 匹配方式 | 键列 | 精品名称（H） |
|---------|------|--------------|
| contains | N | 吉利全系下护板 |
| exact | M | 吉利原厂星瑞贯穿尾灯 |
| contains | N | 吉利全系皮老头豪华脚垫 |
| contains | N | 保养 |

### 5.3 AH 整车超额

依赖已算 **AG、AI、AJ、AK** 与 **D、AA**：

- perf_sum = AG + AI + AJ + AK
- 渠道 D 为分公司/网络部/大客户/含「二网」→ 空白
- 直营店店面 / 自有店 且 AA>0 → `AA × 20%`
- 直营店店面 / 自有店 → `AA × 14%`（与上条分支互斥）
- 其他有效渠道：若 AA×14% + perf_sum < 150 → `150 − perf_sum`

### 5.4 AQ 特殊车型追加绩效

1. `SUMIF(重功超期+活动!E(VIN号), O, 重功超期+活动!N(销售追加奖励))`
2. + `提成标准!H(政策追加) × K`（同 AG 的 C/D/E 三维匹配，用 H/I/A）

## 7. 骨架特殊规则（YAML）

配置路径：`salary_pipeline/config/performance_sheet_columns.yaml`

### 7.1 顾问别名（11 个 VIN）

Hub 汇总时 P 列与 `系统销售毛利!BJ` 不一致的 11 处，由 `p_overrides_by_vin` 覆盖（如袁萍、沈燕1、余才万1/2/3/5）。

### 7.2 补录行（11 行）

月末非 DC 订单号的服务类行（置换服务、爱车保、延保、上户等），`G` 为品类标签而非 DC 订单号，`K` 可为空。补录 VIN 列表见 YAML `supplement_rows`。

## 9. 脱离金标准数值（month.yaml）

```yaml
performance_sheet:
  use_computed: true
  use_golden_perf_sheet: false   # Hub / 销售顾问模块不读金标准整理表数值

hub:
  bootstrap_from_golden: false   # Hub 不回退读金标准提成汇总单元格
```

- **仍读取：** 15 张 input 明细表 + `销售任务及完成率` 等（原始数据，非预计算中间表）
- **仍读取（结构）：** `提成汇总` 的 **店别/职务/姓名** 行键（`SummarySkeletonModule`），不含指标数值
- **对账：** 与金标准 xlsx 比对会有差异（尤其 F–P 中无公式的静态格、销售顾问 W–AI 常数格如韩柏成 Y=1000）
