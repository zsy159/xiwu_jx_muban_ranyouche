# 提成汇总 — 列级计算规则与数据源追溯

**账期样板：** 2026-05  
**架构：** 全面声明式规则引擎——HubMetricsRuleEngine（F–P 全员）+ HubRuleEngine（销售顾问 / 销售主管 W–AI）+ 岗位 overlay 模块；HubFormulaEngine 拓扑逐格回放已从生产路径整体移除（仅保留类定义供 --reconcile 差异标注与售后账套子类使用）  
**最后更新：** 2026-07-07（用户要求所有销售类岗位统一为销售顾问规则，销售主管/
销售助理并入「销售顾问」family、全部 W–AI 列而非仅 W/Y/Z；`SalesSupervisorPerformanceModule`
已删除，见 §2.4 更新说明 + 非一线管理人工填入验证）

## 0. 读本文档前

### 0.1 三层数据流

```text
底层明细表（系统销售毛利、保险明细、销售任务及完成率、各岗位子表…）
        ↓ PerformanceSheetModule（订单级 / 明细 SUMIF）
绩效整理表（按 VIN 行，列 AG/AI/AJ… 系统计算）
        ↓ HubMetricsRuleEngine（F–P 声明式规则，全员按姓名匹配）
        ↓ HubRuleEngine + 各岗位 *_performance 模块（W–AI overlay）
提成汇总（每人一行：F–P 指标 + W–AI 绩效）
        ↓ 发薪引擎 SUMIF
发薪表（XW / 直营店 / CS 提成-发）
```

**原则：** 系统产出 **不从金标准拷贝数值**；对账差异在 `--reconcile` 报告中可见（见 `.cursor/rules/no-golden-bootstrap.mdc`）。

**2026-07-06 决策（用户书面授权）：** 全面废除按 Excel 行号 / topology 公式逐格回放（`HubFormulaEngine.apply()` 在生产 `SalesPipeline.run()` 中的调用）。同一 Hub 列若因店别/姓名不同而取值方式不同，一律在**同一套规则**内用显式分组 / 模板选择器表达（如 H 列完成率封顶分组、W 列 `template_selector`），不再依赖金标准行号推断——生产 topology（合并上传表）本就不含「提成汇总」公式，行号回放对当月数据已是空转。`HubFormulaEngine` 类定义仍保留（售后账套 `AftersalesFormulaEngine` 子类化使用、GUI 字段拉通工具 `aligned_input.py` 的 `detect_template()` 仅供 reconcile 参考），但**不再是销售账套生产计算路径的一部分**。

### 0.2 引擎分工（全面规则引擎后）

| 层级 | 负责引擎 / 模块 | 配置入口 |
|------|-----------------|----------|
| F–P 指标列（全员，按姓名匹配） | **`HubMetricsRuleEngine`** | `config/hub_metrics_rules.yaml` |
| 销售顾问 / 销售主管 / 销售助理 W–AI（同一套规则，2026-07-07 起统一） | **`HubRuleEngine`**（`SalesAdvisorPerformanceModule`） | `config/hub_column_rules.yaml` |
| 新媒体 / 邀约 / 客户 / 直营店经理 / 招聘 W–AI | 各 `modules/*_performance.py` | `config/hub_performance.yaml` |
| 非一线管理 W/X/Y/Z | **人工填入（`delegate: manual_semantic`）**：金标准验证系固定数值/引用人工子表，非公式计算，不作规则/拓扑兜底 | `config/hub_column_rules.yaml` |
| 非一线管理语义列（售后总产值/台次等） | `apply_non_frontline_columns`（不受本次改动影响） | `pipelines/non_frontline_columns.py` |
| 对账差异标注（只读） | `HubFormulaEngine` + topology JSON | `--reconcile` / `build_reconcile_deferred_cells` |

## 1. F–P 指标列（HubMetricsRuleEngine 声明式规则）

由 `HubMetricsRuleEngine.apply()` 按 `config/hub_metrics_rules.yaml` 固定规则计算，**全员适用**（按姓名匹配底层表，未匹配 = 0，与职务/角色无关——出纳/内勤/经理等无任务的人得 0 属正常）。不读取 topology、不依赖金标准行号。销售顾问完成率乘数 **H** 来自本段计算结果。

### 1.1 F / G / H（任务与销量完成率）

| Hub 列 | 表头 | 主要来源表 | 源列 / 逻辑 |
|--------|------|------------|-------------|
| **F** | 考核量 | `销售任务及完成率` | `SUMIF(C, 姓名, Y)` |
| **G** | 实际销量 | `销售任务及完成率` | `SUMIF(C, 姓名, Z)` |
| **H** | 销量完成率 | 派生 | `IF(F≠0, MIN(G/F, 封顶), 0)` |

**H 封顶分组（`hub_metrics_rules.yaml` → `cap_overrides`，按店别固定分组，非行号推断）：**

| 分组 | 店别 | 封顶 |
|------|------|------|
| 默认 | 崇州/大邑/邛崃/彭州/华阳直营店、机场展厅、机场DCC、武侯DCC 等其余店别 | **120%** |
| 覆盖 | 直营店管理、新媒体销售部、西物-翼真、销售管理部 | **110%** |

> 分组来源：2026-05 金标准拓扑只读扫描（遍历 `提成汇总!H{row}` 公式按店别分组统计 cap 取值），仅用于**推导固定分组**，不回填数值、不做逐行推断。若未来账期出现新店别，需用户显式确认应归入哪一分组。

实现：`salary_pipeline/pipelines/hub_metrics_rule_engine.py`（`op: ratio_with_cap_group`）。

### 1.2 I–P（其他指标）

| Hub 列 | 表头 | 规则（`op`） | 来源 |
|--------|------|------|------|
| **I** | 集客达成率 | `lookup_first`（INDEX/MATCH 首个匹配） | `销售任务及完成率!F`，按姓名 |
| **J** | 加装额 | `sumif` | `绩效整理表!S`（computed_perf_frame），按姓名(P) |
| **K** | 加装销量完成率 | `ratio` | `J / (G×1500)`，分母为 0 时取 0 |
| **L** | 保险渗透率 | `filtered_ratio` | `绩效整理表`：`SUMIFS(K, AB>0, P=姓名) / SUMIF(P=姓名, K)` |
| **M** | 整车毛利 | `sumif` | `绩效整理表!BG`，按姓名(P) |
| **N** | 加装毛利 | `sumif` | `绩效整理表!BI`，按姓名(P) |
| **O** | 保险毛利 | `sumif` | `绩效整理表!AB`，按姓名(P) |
| **P** | 按揭毛利 | `sumif` | `绩效整理表!AC`，按姓名(P) |

`绩效整理表` 均指 `computed_perf_frame`（`PerformanceSheetModule` 系统计算，非金标准）。规则配置：`config/hub_metrics_rules.yaml`；引擎实现：`hub_metrics_rule_engine.py`。

2026-05 金标准回归验证（`tests/test_hub_metrics_rule_engine.py::HubMetricsRuleEngineGoldenParityTest`）：唐鹏、赵思梵两名顾问 F–P 十一列与金标准逐列一致；余才万（直营店管理，35/40=114.3%）验证 110% 分组确实生效（若误用默认 120% 上限则不会封顶）；牟春柳（无任务表数据）验证全 0 属预期而非缺陷。

## 2. 完成率乘数（销售顾问 / 销售主管）

销售顾问、销售主管 W/Y/Z 使用完成率时，乘数来源如下。

### 2.1 销量完成率 H（个人块 + 加装/保险）

| 项目 | 说明 |
|------|------|
| 写入位置 | 提成汇总 **H 列**（F–P 阶段已算好） |
| 用于 | 个人块 **W**；全店 **Y、Z**；`store_ba` 模板的加装/保险 |

### 2.2 合并完成率 BA（门店块整车专用）

| 项目 | 说明 |
|------|------|
| 来源表 | `销售任务及完成率` |
| 源列 | **AG**（总完成率 / 合并完成率） |
| 匹配键 | **C 列姓名**（`lookup_combined_completion_rate`） |
| 系统封顶 | **无**（表内 120% 等原样使用） |
| 用于 | 仅 **W 整车绩效**（`store_ba` 模板） |

### 2.3 店别模板选择（`hub_column_rules.yaml`）

| 模板 | 判定 | W | Y / Z |
|------|------|---|-------|
| `store_ba` | 店别 ∈ `store_ba_shops` | `SUM(AG)×BA` | `SUM(AI/AJ)×H` |
| `personal_h` | 默认 | `SUM(AG)×H` | `SUM(AI/AJ)×H` |
| `insurance_add` | 姓名覆盖（韩柏成） | 同 `personal_h` | `SUM(AJ)×H + 600` |

**`store_ba_shops`（2026-05 金标准扫描）：** 崇州直营店、大邑直营店、邛崃直营店、彭州直营店、华阳直营店。

> 2026-05 样本月 `提成汇总` B 列店别中 **无「紫荆山直营店」**；文档历史举例勿与当月数据混用。

### 2.4 销售主管 / 销售助理：并入销售顾问统一规则（2026-07-07 更新）

**2026-07-07 用户明确要求：** 所有销售类岗位（销售顾问/销售主管/销售助理）都按
销售顾问同一套规则计算（含 `template_selector` 的 `store_ba`/`personal_h` 判定），
"后面有问题再改"。原独立 `销售主管` role_family（W/Y/Z 恒定 ×H、不设
`template_selector`）与 `modules/sales_supervisor_performance.py` 已删除；
`config/hub_column_rules.yaml` 的 `销售顾问` family `match.职务` 现为
`[销售顾问, 销售主管, 销售助理]`，统一由 `SalesAdvisorPerformanceModule`
（`match_advisor_row`）写 overlay。销售助理此前未登记任何 Hub 列，现一并纳入。

**历史依据（此前 2026-05 金标准只读验证，供复查差异用）：** 销售主管
（职务=销售主管）W/Y/Z 完成率乘数原判定**恒定为 H**（个人销量完成率），不使用
`store_ba` 模板的合并完成率 BA。验证样本 `提成汇总` row 60 邓戈/机场展厅、
row 107 熊杰文/武侯DCC，两人门店均**不在** `store_ba_shops` 清单内，故并入统一
规则后 `personal_h` 兜底结果与原 H-only 规则数值一致，W/Y/Z 三列未受影响；新增
的其余列（金融/爱车宝/上户绩效等，此前未登记）现随统一规则一并计算。

**已知权衡：** 若未来有销售主管/助理落在 `store_ba_shops` 门店，将按 `store_ba`
模板切到合并完成率 BA，与本条历史验证的「恒定 H」结论不同——用户已知悉此权衡，
选择先统一简化，出现问题再单独处理。

## 3. 销售顾问 W–AI（HubRuleEngine）

**规则文件：** `salary_pipeline/config/hub_column_rules.yaml`  
**实现：** `salary_pipeline/pipelines/hub_rule_engine.py`  
**接入：** `salary_pipeline/modules/sales_advisor_performance.py`  
**汇总键：** `绩效整理表!P` = 顾问姓名

### 3.1 Hub 列 → 整理表 → 底层表（总表）

| Hub 列 | 表头 | Hub 计算 | 整理表源列 | 底层追溯（明细 / 闭包） |
|--------|------|----------|------------|-------------------------|
| **W** | 整车绩效 | `SUM(AG)×H` 或 `×BA` | **AG** | 见 §3.2 |
| **X** | 权限结余绩效 | `SUM(AH)` | **AH** | 见 §3.3 |
| **Y** | 加装绩效 | `SUM(AI)×H` | **AI** | 见 §3.4 |
| **Z** | 保险绩效 | `SUM(AJ)×H`（+常数） | **AJ** | `保险明细` D→**BS** |
| **AA** | 金融绩效 | `SUM(AK)` | **AK** | `按揭明细` G→**BO** |
| **AB** | 爱车宝绩效 | `SUM(AM)` | **AM** | `爱车保` F→**BA** |
| **AC** | 上户绩效 | `SUM(AN)+SUM(AS)` | **AN, AS** | `上户提成` B→H；`置换服务` G→BB |
| **AD** | 盈利产品绩效 | `SUM(AL)` | **AL** | `按揭原表` AC→AF + `按揭明细` G→BR |
| **AE** | 延保提成 | `SUM(AT)` | **AT** | `延保提成` F→BE（±200 规则） |
| **AF** | 特殊车型+指定车型 | `SUM(AQ)` | **AQ** | `重功超期+活动`、`提成标准`、`系统销售毛利` 等 |
| **AG** | 座位险提成 | `SUM(AO)` | **AO** | `保险明细` D→**BU** |
| **AH** | 二手车提成 | `SUM(AR)` | **AR** | `二手置换`、`大客户` |
| **AI** | 玻碎险提成 | `SUM(AP)` | **AP** | `保险明细` D→**BV** |

登记与测试列（`hub_performance.yaml` parity_gate）：W、Y、Z、延保、特殊车型、金融、爱车宝、上户等 8 列核心绩效。

### 3.2 整理表 AG（单台绩效）— 整车链

```text
VIN（O）← 系统销售毛利!BD
    ├─ D/H 车种、E/I 渠道、BL→比对表→部门 A
    └─ K 台数

查表：上传账套「提成标准」sheet（≈ 提成依据·销售提成标准）
    键：C(店别) × D(渠道) × E(车种) → F「标准」单台基数

AG = F × K
特例：A=武侯自有店 且 H=星越L → 200×K（不走 lookup）
```

代码：`calculators/performance_sheet/from_closure._compute_ag`  
配置：`config/performance_sheet_columns.yaml` → `closure_columns.AG`

### 3.3 整理表 AH（权限结余）

闭包列，非简单 SUMIF。由 `系统销售毛利` 权限列 AA 与渠道标签（A+I 派生）分支：

- 直营店店面 / 自有店 且 AA>0 → AA×20%
- 直营店店面 / 自有店 → AA×14%
- 其他分支见 `from_closure._compute_ah`

Hub **X** = 按 P 汇总 AH，**不乘** H/BA。

### 3.4 整理表 AI（加装）

闭包列：`装饰台账`、`系统销售毛利` 等组合（`closure_columns.AI`）。  
Hub **Y** = `SUM(AI)×H`（门店块 W 用 BA 时，Y 仍用 H）。

### 3.5 整理表明细 SUMIF 列（直引底层表）

摘自 `performance_sheet_columns.yaml`：

| 整理表列 | 底层表 | 键列 | 值列 |
|----------|--------|------|------|
| AJ | 保险明细 | D (VIN/O) | BS |
| AK | 按揭明细 | G | BO |
| AO | 保险明细 | D | BU |
| AP | 保险明细 | D | BV |
| AL | 按揭原表 + 按揭明细 | AC / G | AF / BR |
| AM | 爱车保 | F | BA |
| AN | 上户提成 | B | H |
| AS | 置换服务 | G | BB |
| AT | 延保提成 | F | BE |
| AQ | 多源闭包 | — | — |
| AR | 二手置换 + 大客户 | — | — |

绩效整理表 **P 列**（顾问）多数来自 `系统销售毛利!BJ`，部分 VIN 有 `p_overrides_by_vin` 别名覆盖。

## 5. 发薪表如何读 Hub

发薪引擎 **只读** `output/<月>/提成汇总.xlsx`（`payout.use_computed_hub: true`）。

| 发薪列（XW 示例） | 典型 SUMIF |
|-------------------|------------|
| H 整车绩效 | `SUMIF(提成汇总!D, 姓名, 提成汇总!W)` |
| I 权限结余 | `SUMIF(..., X)` |
| J/K 加装 | `SUMIF(..., Y)` |
| … | W–AI 同名列 |

**例外：** 崇州邀约 / 直营店经理 → Hub **AK** → 发薪 **V**；招聘 → Hub **Z** → 发薪 **K**。

详见 [发薪表-岗位取数与计算逻辑.md](../payout/发薪表-岗位取数与计算逻辑.md)。

| 非一线管理人工填入登记 / 手工高亮列 | `config/hub_column_rules.yaml`（`delegate: manual_semantic`）+ `pipelines/commission_summary_column_sources.py` |
| 客户关系部不应归入非一线 support tier | `config/non_frontline_roles.yaml`（`support.shops`，2026-07-07 移除，见 §Part B 客户专员根因） |
| 对账差异标注（只读，非生产计算） | `data/topology/<月>/*.topology.json` + `hub_formula_engine.py` |

### 运行与增量

```bash
# 全量（改上传数据 / 绩效配置 / F–P 规则 hub_metrics_rules.yaml）
python main.py compute --reconcile

# 仅改销售顾问规则或计算器
python main.py compute --from hub --only sales-advisor --reconcile
```

观察台：`python run_console.py` → 上传 → 试算 → 查看 `output/<月>/.staging/提成汇总.xlsx`。
